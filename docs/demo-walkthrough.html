Demo walkthrough — one command, five steps

python demo/full.py runs the whole read-only pipeline against the synthetic GreenLeaf Landscaping account and fills all five Sheet tabs. This page is the mechanics of each step — what runs, what it reads, what it writes — for my own reference while demoing or debugging.

Before you run

One command

Everything below happens in sequence from this single command. Re-running is safe — Step 1 wipes the tabs first, so every run is a clean slate.

This page is for me, not the boss. The client-facing pitch is visual-guide.html. This one is the working mechanics, and it does write rows to the live Google Sheet (all tagged SYNTHETIC DEMO DATA - safe to delete).

$ python demo/full.py

Reuses the real tools/ code throughout — detection, scoring, the grounding gate, the idempotent Sheet upsert. Nothing here re-implements that logic. A ~2 s pause between writes keeps it under the Sheets ~60-req/min quota (with 429 back-off as a safety net).

The pipeline, step by step

What happens, in order

Each card: what the step does mechanically, what it reads & writes, which tab it fills, and the exact console output from a real run.

1

Reset all demo tabs

reset.reset_all(svc, sid) clears all 5 tabs

Clears every data row (row 2 onward) from the five demo tabs and keeps the header row, so the run starts from a clean slate. Tab names resolve from sheet_schema.TARGETS (env-aware), so they never drift.

Deliberately leaves the _state tab alone — that holds the durable last-seen cursor; wiping it would make Layer 1 re-process or skip Requests.

readseach tab's current rows (for the count)
writesvalues().clear() on A2:Z — reversible

Walks the 7 GreenLeaf Requests. For each, it reads the linked client and runs the real rules: completeness() for missing fields and find_duplicates() for duplicate signals (the live Jobber search is stubbed to the fixture account).

Every Request gets a Requests row; any flagged client also gets a Client Hygiene row. Finds: one strong dup (shared phone), one possible dup (same name+address), one incomplete.

3

Score + prioritize leads

scoring_rules.score_lead(req, client) Prioritized Leads

Scores 8 inbound leads 0–100 from five signals — source, job value, urgency, service-area fit, reachability — then sorts highest-first and writes one ranked row per lead. The suggested tier is hot ≥75 · warm ≥55 · cool ≥35 · else defer.

Scoring is internal triage — it never leaves the Sheet, so it does not go through the grounding gate.

4

Generate follow-up drafts

hot_draft(req) · scoring.deferral_draft(req) → check_draft_grounding.py Follow-up Drafts

Drafts only where it makes sense: hot → a personalized reply; defer + reachable → a warm waitlist note. Warm/cool are skipped; a defer lead with no contact info can't be reached, so it's skipped too.

Every draft passes through the real grounding gate. Claims are limited to what's confirmed (free estimate → pricing.md; tone → voice.md), so each lands status=drafted — never an invented price or promise.

Cleanup proposal for the strong duplicate

find_duplicate_clients.find_duplicates → propose_cleanup.propose Cleanup Proposals

Finds the strong-duplicate pair (John Smith / J. Smith, shared phone) and builds a high-confidence-only merge_duplicates proposal: keep the most complete record as primary, carry over the sibling's real values (here, the second email). Nothing is invented; nothing is executed — the merge itself is Layer 3.
